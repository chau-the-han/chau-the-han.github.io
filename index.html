    Vague questions, guessed answers

    Staff ask in everyday language without dates or scope — the AI picks an interpretation and runs SQL without checking.

    This week 42

    Security & Tenant Isolation

    Letting AI query your database sounds risky — here's exactly how we keep it safe.

    02 · The Core Idea

    Can the AI Run SQL on Our Database?

    No — the model only proposes SQL. Our app decides what runs: every query is validated and executed read-only on your clinic's database.

    02 · Defense in Depth

    🛡️ Six Independent Safety Layers

    Every layer must pass before a single row is returned — even if one safeguard were bypassed, the others still protect your data.

    1🏢 One database per clinicPhysical separation — Clinic A and Clinic B never share tables or storage.App connects to clinic_a_db — not clinic_b_db
    2🔒 Read-only credentialsThe assistant's DB user has SELECT permission only — no INSERT, UPDATE, or DELETE.Even if DELETE slipped past the parser, the DB would refuse it
    3🧬 SELECT-only parser gateEvery query is parsed as an AST; anything but a single SELECT is rejected.DELETE, DROP, TRUNCATE, UPDATE → all blocked at parse time
    4📋 Table allowlistOnly 5 approved clinic tables; system catalogs and internals are blocked.patient_users, reservations, … — not pg_tables or information_schema
    5🙈 Sensitive-column blocklistPasswords, hashes, and encrypted fields can never be selected.password, email_hash, token columns → auto-detected and blocked
    6✂️ Row cap & soft-delete filterAutomatic LIMIT 100; deleted records require explicit deleted_at IS NULL.No runaway full-table dumps; archived rows stay hidden
    02 · Blocked Examples

    ❌ What the SQL Gate Rejects

    Even if the AI tries something dangerous, the gate stops it before it touches your data.

    ❌ Blocked — destructive command
    💬 "Delete all cancelled reservations"
    DELETE FROM reservations WHERE status = 90
    🚫 Rejected: Only SELECT statements are allowed
    ❌ Blocked — hidden system tables
    💬 "Show me all database tables"
    SELECT * FROM pg_catalog.pg_tables
    🚫 Rejected: Table not on allowlist (only 5 clinic tables permitted)
    ❌ Blocked — sensitive columns
    💬 "List patient passwords"
    SELECT password, email_hash FROM patient_users
    🚫 Rejected: Column not allowed (passwords & hashes are blocked)
    02 · Allowed Example

    ✅ What the SQL Gate Approves

    A normal analytics query that passes all checks and runs safely on the read-only database.

    ✅ Allowed — normal analytics
    💬 "How many reservations today?"
    SELECT COUNT(*) FROM reservations WHERE deleted_at IS NULL AND start_at::date = CURRENT_DATE LIMIT 100
    ✅ Runs safely — soft-delete filter enforced, LIMIT auto-added
